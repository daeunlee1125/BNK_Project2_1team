## 기술 스택 & 선택 이유 (Why)

### Backend

- **Java 21**
  - Virtual Threads 도입으로 다중 외환 조회·거래 요청 환경에서  
    동시성 처리 성능 향상 가능성 고려
  - 금융 서비스에 적합한 안정성과 LTS 기반 운영 환경 확보

- **Spring Boot 3.5.x**
  - 금융 트랜잭션 및 보안(Spring Security)에 검증된 프레임워크
  - REST API 중심의 명확한 계층 구조 설계에 유리

- **MyBatis**
  - 금융 도메인 특성상 정밀한 SQL 제어 및 튜닝 필요
  - 복잡한 조회 쿼리를 명시적으로 관리하기 위해 채택

---

### AI / Data

- **FastAPI (Python)**
  - LangChain, KoBART 등 Python 기반 AI 라이브러리와의 높은 호환성
  - 비동기 처리 기반으로 빠른 AI 추론 응답 속도 확보
  - 메인 서비스와 분리하여 장애 전파 최소화

- **FAISS (Vector DB)**
  - 금융 약관·상품 설명서에 대한 의미 기반 검색(RAG) 구현
  - 문맥 유사도 중심의 정확한 답변 제공

- **KoBART**
  - 한국어 뉴스 요약에 특화된 모델
  - 금융 기사 특유의 장문 텍스트 요약 품질 확보

---

### Search / Cache

- **Elasticsearch**
  - 금융 약관 및 뉴스 데이터의 고속 검색
  - 형태소 분석을 통한 정확한 정보 추출
  - 단순 DB 조회 대비 검색 성능 및 확장성 우수

- **Redis**
  - 인증 보조 데이터 및 번역 결과 캐싱
  - 외부 API 호출 비용 절감 및 응답 속도 개선

---

### DevOps

- **GCP + Docker**
  - 인프라 환경 일관성 확보 및 배포 자동화

- **GitHub Actions + Workload Identity Federation**
  - 서비스 계정 키를 사용하지 않는 Keyless CI/CD
  - 배포 시에만 유효한 단기 토큰 사용으로 보안 강화

## 핵심 기능

### 1️⃣ AI 기반 경제·환율 브리핑 & QnA

- 주요 경제지 뉴스 하루 3회 자동 수집

### 2️⃣ 관리자용 AI PDF 위규 분석 도구

- 의미 기반 검색 + 리랭킹을 통해
  - 위반 가능 문구
  - 개선 제안 AI 코멘트 자동 생성

### 3️⃣ 보안 및 성능 최적화 (담당 영역)

- Redis 캐싱을 통한 외부 API 비용 절감

## Troubleshooting

### 1️⃣ CI/CD 보안 이슈 – WIF 도입

- **문제**
  - 기존 GCP 서비스 계정 키(JSON) 방식은 키 노출 시 보안 사고 위험 존재

- **해결**
  - GCP Workload Identity Federation 적용
  - GitHub Actions ↔ GCP 간 Keyless 인증 구조 설계
  - IAM 정책 세분화 및 공식 문서 기반 설정 검증

- **결과**
  - 장기 키 완전 제거
  - 금융 서비스에 적합한 배포 보안 수준 확보

---

### 2️⃣ 환율 API 장애 대응 – Fallback 로직

- **문제**
  - 비영업일 및 외부 API 장애 시 환율 조회 실패

- **해결**
  - 최대 5영업일 이전 환율을 자동 조회하는 Recursive Fallback 로직 설계

- **결과**
  - 외부 장애 상황에서도 서비스 가용성 유지
  - 사용자 체감 오류 최소화
